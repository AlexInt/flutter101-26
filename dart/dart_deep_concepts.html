Dart Event Loop 工作原理

理解 Event Loop 是掌握 async/await、Stream、Zone 的基础

核心规则：Dart 是单线程的，所有代码都在 Event Loop 中执行。Event Loop 优先处理 Microtask Queue，然后处理 Event Queue。

关键理解：

  • Future 的回调进入 Event Queue
  • Future.microtask() / scheduleMicrotask() 进入 Microtask Queue
  • Microtask 优先级高于 Event（每次 Event Loop 循环先清空 microtask）
  • async/await 本质上是把后续代码放入 Microtask Queue

Mixin 线性化（Mixin Linearization）

Dart 特色概念：理解 with A, B 的执行顺序与 super 调用链

核心规则：class C with A, B 的线性化顺序是 C → B → A → Object（从右往左叠加，最右边优先级最高）。调用 super 时沿着这个链向上找。

Zone 异步执行区域

拦截异常、注入变量、自定义异步行为

Zone 是什么？ Zone 是异步回调的执行环境。每个异步操作（Future、Timer、Stream）都在创建它的 Zone 中执行。Zone 可以"捕获"未处理的异常、拦截 print、注入变量。

Zone 嵌套结构

子 Zone 继承父 Zone 的规则，可覆盖特定行为

Root Zone（根 Zone） print: 默认输出到控制台
runZonedGuarded Zone onError: 捕获所有未处理异常 print: 添加 [APP] 前缀
子 Zone（runZoned） values: { userId: '42' } 继承父 Zone 的 onError print: 添加 [DEBUG] 前缀

✅ 此 Zone 中的异步异常 → 被 runZonedGuarded 的 onError 捕获

✅ 此 Zone 中可以访问 userId 变量

✅ print 输出：[DEBUG] 原始内容

🚨 未被 try-catch 捕获的异常 → 会冒泡到这里被 onError 拦截

Flutter 中的标准用法

main.dart 全局错误捕获模板

void main() {
  // Widget 层错误（build 阶段）
  FlutterError.onError = (details) {
    Crashlytics.instance.recordFlutterError(details);
  };

  // 异步错误（Zone 未捕获的）
  runZonedGuarded(() {
    runApp(MyApp());
  }, (error, stackTrace) {
    Crashlytics.instance.recordError(error, stackTrace);
  });
}

为什么不用反射（Reflection）？

  • Dart 的 dart:mirrors 已被废弃，Flutter 直接禁用
  • 反射会阻止 tree shaking → 包体积暴增
  • 代码生成是编译时的 → 零运行时开销 + 编译时检查

Tree Shaking 原理

编译器自动移除未使用代码，减小包体积

原理：从入口（main 函数）开始，编译器追踪所有被引用的代码路径。没有被任何路径引用到的代码会被"摇掉"（不打包到最终产物中）。

Tree Shaking 可视化演示

class MathUtils {

Tree Shaking 的敌人

模式为什么阻止 Tree Shaking解决方案
反射 (dart:mirrors)编译器无法静态分析运行时调用用代码生成替代（Flutter 已禁用 mirrors）
dynamic 滥用类型不确定，需保守保留所有可能使用具体类型
字符串动态查找通过字符串引用无法追踪使用直接引用
